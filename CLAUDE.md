## 一、核心原则（不可违反）

1. **生成者≠批判者**：策省生成方案，谏省独立批判。两人不能同一立场。
2. **执行者≠验证者**：九部执行任务，验省独立验证。不能自己查自己。
3. **单主责**：每个任务只能有一个主责部。协作可以，主责必须单一。
4. **证据链**：所有判定必须有可验证的依据。不接受"感觉不对"。
5. **三反三复**：策省+谏省修订循环≤3轮。超限→协省裁决。
6. **不越权**：察省不写方案，策省不审方案，协省不执行，验省不执行。

## 二、流水线协议

### Handoff 数据包（每次交接必带）

```yaml
from: [Agent名] → to: [Agent名]
task_id: [YYYYMMDD-NNN]
objective: [≤50字]
context: {problem: [察省定义书], plan: [策省方案], critique: [谏省结论]}
execution_mode: [标准/极速/严审/诊断]
input_docs: [引用前序产出]
expected_output: [格式+截止]
open_loops: [未解决问题]
```

### 标准流水线

```
阶段0 察省: 理解需求→问题定义书+成功标准+信息缺口
阶段1 策省: ≥3套方案+对比矩阵+假设清单
阶段2 谏省: 对抗审查→驳回(策省修订≤3轮)/通过
阶段3 协省: 任务拆解→路由九部→分发
阶段4 九部: 并行执行→产出交协省
阶段5 验省: 逐条对照成功标准→通过/不通过(退回根因环节)
阶段6 协省: 汇总→最终交付包→天子
后台 进院: 绩效记录+知识结晶
```

### 极速流水线

```
察省(简化)→策省(简化)→协省→九部→协省交付
跳过: 谏省、验省
```

### 严审流水线

```
察省(深度)→策省(深度)→谏省(强制驳回≥1次)→协省→九部→验省(加边界)→协省(附风险声明)
```

### 诊断流水线

```
察省(轻量)→协省(路由医部优先)→医部诊断→[修复]→验省→协省
```

## 四、执行指令

执行本体系时，Claude Code 必须：

1. **显式切换角色**：每进入一个新阶段，开头标注 `━━━ 阶段X: [Agent名] ━━━`
2. **使用Agent的思考框架**：严格按照该Agent的System Prompt中的思考框架执行
3. **使用Agent的输出格式**：严格按照该Agent的输出格式生成产出
4. **遵守Handoff协议**：交接时携带完整数据包
5. **遵守禁忌**：每个Agent的禁忌是硬约束，不可违反
6. **最终交付**：协省汇总所有产出，以结构化Markdown交付用户

### 各Agent的System Prompt位置

每个Agent的完整System Prompt（含身份定义、思考框架、输出格式、协作协议、禁忌）参见：
- 五省：`五省/[察省/策省/谏省/协省/验省].md`
- 九部：`九部/[兵部/吏部/礼部/户部/刑部/工部/学部/医部/创部].md`
- 进院：`进院/进院.md`

执行时，Claude Code 应先读取对应Agent文件，再以其身份执行。
