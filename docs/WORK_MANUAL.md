## 〇、第一性原理

### 体系是什么

```
591 = 五省(5) + 九部(9) + 一院(1) = 15 个 Agent 角色

输入：天子的一句话需求
输出：结构化交付包
过程：察→策→谏→协→九部执行→验→协交付→进院记录
```

### 体系不是什么

- 不是代码，不是 API，不是后台服务
- 是 **15 个 AI 角色的 System Prompt 集合 + 流水线协议**
- 运行在单个 Claude Code 会话中，通过「角色扮演」模拟多 Agent 协作
- 代理之间的独立性是「文本角色切换」，不是「进程隔离」

### 核心不变原则（7 条）

| # | 原则 | 违反后果 |
|---|------|---------|
| 1 | 生成者 ≠ 批判者 | 确认偏差 → 错误方案不被发现 |
| 2 | 执行者 ≠ 验证者 | 盲区 → 执行偏差不被发现 |
| 3 | 单主责 | 同一任务分给两个部 → 责任模糊 |
| 4 | 证据链 | 「感觉不对」→ 无法追溯决策依据 |
| 5 | 三反三复 | 策省+谏省循环 ≤3 轮 → 升级协省裁决 |
| 6 | 不越权 | 察省不写方案，策省不审方案，协省不执行 |
| 7 | 诚实边界 | 信息不足 → 标注缺口，不编造 |

## 二、加载方式

### 方式 1：斜杠命令（推荐）

在 Claude Code 中直接输入：

```
/591 帮我设计一个新产品上市策略
/591 极速 帮我给这段代码加注释
/591 严审 帮我校验这份投资方案的风险
/591 诊断 网站访问速度突然下降，帮我排查
/591 进化 回顾上个月的任务，优化系统
```

Claude Code 自动读取 `.claude/commands/` 下的命令文件，按流水线执行。

### 方式 2：直接加载 CLAUDE.md

如果斜杠命令未注册或你希望手动控制：

```
Read: 591/CLAUDE.md
然后说："用 591 标准模式处理以下任务：[任务描述]"
```

### 方式 3：单 Agent 直接调用

如果只需要某个特定 Agent 的能力，不跑完整流水线：

```
Read: 591/九部/医部.md
然后说："以医部身份诊断：[问题描述]"
```

## 三、流水线操作

### 3.1 标准流水线（/591）

```
天子 → 察省 → 策省 → 谏省 → 协省 → 九部(并行) → 验省 → 协省交付 → 进院
```

| 阶段 | Agent | 操作 | 产出 |
|------|-------|------|------|
| 0 | 察省 | Read 察省.md → 分析需求 | 问题定义书 |
| 1 | 策省 | Read 策省.md → 生成方案 | ≥3套方案 + 假设矩阵 |
| 2 | 谏省 | Read 谏省.md → 对抗审查 | 审议意见（驳回/通过） |
| 3 | 协省 | Read 协省.md → 拆解路由 | 任务分派 |
| 4 | 九部 | 按需 Read 对应部文件 → 执行 | 各部交付物 |
| 5 | 验省 | Read 验省.md → 逐条验证 | 验证报告 |
| 6 | 协省 | 汇总交付 | 最终交付包 |
| 后台 | 进院 | Read 进院.md → 绩效记录 | 知识结晶 |

### 3.2 极速流水线（/591 极速）

跳过谏省和验省。**仅用于低风险、熟悉的任务。**

```
察省(简化) → 策省(简化) → 协省 → 九部 → 协省交付
```

### 3.3 严审流水线（/591 严审）

谏省强制驳回 ≥1 次，验省加边界测试。**用于高风险决策。**

```
察省(深度) → 策省(深度) → 谏省(强制驳回 v1→v2→v3) → 协省 → 九部 → 验省(边界测试) → 协省(风险声明)
```

### 3.4 诊断流水线（/591 诊断）

医部优先，用于故障排查。

```
察省(轻量) → 协省(路由医部) → 医部(6步诊断框架) → [修复] → 验省 → 协省
```

## 四、Handoff 协议

每次 Agent 之间交接时，必须携带以下数据包：

```yaml
from: [Agent名] → to: [Agent名]
task_id: [YYYYMMDD-NNN]
objective: [≤50字]
context: {problem: [察省定义书], plan: [策省方案], critique: [谏省结论]}
execution_mode: [标准/极速/严审/诊断]
input_docs: [引用前序产出]
expected_output: [格式+截止]
open_loops: [未解决问题]
```

**操作要点：**
- 每个 Agent 产出末尾标注 `---` 和 `请[下一个Agent]基于此[...]`
- 协省负责检查数据包完整性
- 验省负责验证最终产出是否满足原始成功标准

| 重叠领域 | 分工 |

| 市场调研 | 学部做「行业深研」→ 兵部做「竞品矩阵」 |
| 安全/合规 | 刑部做「事前预防」→ 医部做「事后诊断」 |

## 六、配置与定制

### 6.1 修改 Agent 提示词

每个 Agent 的 System Prompt 独立存储在对应的 `.md` 文件中。

**修改流程：**
1. 找到对应文件（如 `五省/察省.md`）
2. 编辑代码块内的 System Prompt
3. 遵守 Agent 的文件格式：角色定义 → 第一性原理 → 思考框架 → 输出格式 → 协作协议 → 禁忌
4. 更新文件头部的版本号
5. 运行一次测试任务验证

### 6.2 新增执行模式

在 `.claude/commands/` 下创建新的 `.md` 文件。文件内容 = 执行步骤描述 + Read 指令。

### 6.3 注册新部门（第十部）

当某个问题域频繁被标记为「领域缺口」时：

1. 创部 + 学部 生成临时 Agent 配置（名称 + System Prompt + 技能依赖）
2. 经过 ≥3 次成功使用后，进院评估是否应注册为永久部门
3. 创部生成正式 Agent 配置文件，放入 `九部/` 目录
4. 更新 `CLAUDE.md` 的 Agent 清单表、路由表、复合任务表

### 6.4 API Token 配置

591 体系本身不包含 API 调用（所有 Agent 是角色扮演）。但九部中的工部、学部、医部在执行任务时可能需要外部 API：

- 工部：Cloudflare API（需要 Token，5 行权限）
- 医部：Cloudflare API（安全扫描）
- 学部：Web Search API

这些 Token 由 Claude Code 的 settings.json 管理，不存储在 591 目录中。

## 七、调试与故障排除

### 7.1 常见问题

| 症状 | 可能原因 | 解决 |
|------|---------|------|
| 所有九部输出相同 | 未切换 Agent Prompt | 确认每个部执行前 Read 了该部的 .md 文件 |
| 谏省从不驳回 | 审查太松 | 检查谏省.md 中的「硬性指标」（驳回率 ≥50%） |
| 流水线中断 | 上下文窗口不足 | 切换到极速模式或拆分为多个会话 |
| Agent 之间产出不一致 | Handoff 数据包缺失 | 检查协省是否正确携带了 context |
| 验省总说「通过」 | 验省未独立验证 | 验省必须独立核对，不能直接接受执行部的自我报告 |
| 某个部不知道该用什么工具 | 技能依赖未加载 | 检查该部 .md 文件的「技能依赖」章节 |

### 7.2 调试模式

在 Claude Code 中调试单个 Agent：

```
# 不用完整流水线，直接调用单个部
Read: 591/九部/医部.md
Read: 591/CLAUDE.md （只看路由部分）
以医部身份，仅执行诊断，不进入完整流水线：[问题描述]
```

### 7.3 上下文预算

完整流水线（标准模式）的大致 token 消耗：

| 阶段 | Token 估算 |
|------|-----------|
| CLAUDE.md 加载 | 2K |
| 察省 | 2K（Prompt）+ 1K（输出） |
| 策省 | 2K + 3K |
| 谏省 | 1.5K + 2K |
| 协省 | 1.5K + 1K |
| 九部执行 | 2K/部 × 参与部数 |
| 验省 | 1.5K + 1K |
| **合计（3部参与）** | **约 25K tokens** |

如果上下文不足，使用极速模式可节省约 40%。

## 八、版本升级

### 从任一旧版本升级到 v2.3

1. `git pull origin main`
2. 检查 `CHANGELOG.md` 了解变更
3. 如有不兼容变更，参考 `docs/MIGRATION.md`
4. 重新加载 `CLAUDE.md`（新版本号会自动生效）

### 版本号规则

- 主版本号（vX）：架构变更（如三省六部 → 五省九部一院）
- 次版本号（vX.Y）：新增 Agent/模式/章节
- 修订号（vX.Y.Z）：错误修复/措辞修订

## 九、诚实边界

591 不是万能的。以下情况下体系会诚实告知限制：

| 限制 | 行为 |
|------|------|
| 信息不足 | 察省标注信息缺口，请求用户补充 |
| 资源不足 | 策省标注资源需求，请求批准 |
| 套餐/权限限制 | 降级处理（如 Free 套餐不请求 Pro 功能） |
| 上下文不足 | 建议拆分为多个会话 |

详见 `docs/HONEST_CAPABILITY_BOUNDARIES.md`。
