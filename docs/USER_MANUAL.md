## 〇、三十秒理解 591

```
你说一句话 → 591 自动完成完整思考→方案→审查→执行→验证→归档

你的角色：天子（定愿景、批准关键决策、承担最终责任）
591的角色：15个AI顾问团队（分析、提案、反驳、执行、验证、进化）
```

### 它像什么

| 类比 | 说明 |
|------|------|
| 一个顶级参谋团队 | 有人分析问题、有人给方案、有人挑刺、有人执行、有人验收 |
| 一个自动化的 PDCA 循环 | Plan（察省+策省）→ Do（九部）→ Check（谏省+验省）→ Act（进院） |
| 一个永不疲倦的顾问委员会 | 24/7 可用，不会因为「太忙」跳过审查环节 |

### 它不像什么

- 不是一个聊天机器人（它有 15 个独立角色，有流程纪律）
- 不是一个代码生成器（虽然能写代码，但核心是解题方法论）
- 不是一个万能工具（它诚实标注自己能做什么、不能做什么）

## 一、快速开始

### 1.1 四个核心命令

在 Claude Code 中输入：

```
/591 [你的任务]
```

**四种模式**：

| 命令 | 什么时候用 | 示例 |
|------|-----------|------|
| `/591` | 重要决策、复杂任务 | `/591 设计新产品的上市策略` |
| `/591 极速` | 简单、熟悉的小任务 | `/591 极速 给函数加注释` |
| `/591 严审` | 高风险、不可逆的决策 | `/591 严审 审核这份投资方案` |
| `/591 诊断` | 出问题了，需要排查 | `/591 诊断 网站流量下降50%` |

### 1.2 第一个例子

```
/591 帮我做一个竞品分析：我的产品是5加仑PC饮水桶，目标市场是中东
```

591 会自动：
1. **察省**：先问清楚你的当前状态、期望目标、预算约束
2. **策省**：给出 3 套不同的竞品分析方案（保守/激进/创新）
3. **谏省**：挑出每套方案的致命缺陷（如「激进方案需要的中东当地数据可能拿不到」）
4. **协省**：把任务拆给兵部（市场）+ 学部（中东研究）+ 户部（定价核算）
5. **九部执行**：各部产出自己的交付物
6. **验省**：逐条验证是否达到了察省定义的成功标准
7. **交付**：一份完整报告，含竞品矩阵 + SWOT + 定价建议 + 风险清单

## 二、怎么用好 591

### 2.1 输入越具体，输出越有价值

| 差的输入 | 好的输入 |
|---------|---------|
| 「帮我写个方案」 | 「帮我写一个产品的 Go-to-Market 方案：产品是XX，目标市场是XX，预算是XX，时间窗口是X个月」 |
| 「网站慢了」 | 「网站过去7天加载时间从2秒变成了5秒，影响北美用户，不确定是CDN问题还是代码问题」 |
| 「帮我招人」 | 「招一个远程的React前端，3-5年经验，预算是X/月，需要熟悉Next.js和Tailwind」 |

### 2.2 选对模式

| 场景 | 模式 | 为什么 |
|------|------|--------|
| 战略规划 | `/591` 标准 | 需要多方案比较和严格审查 |
| 改个文案 | `/591 极速` | 低风险，不值得跑完整审查 |
| 签合同前审查 | `/591 严审` | 法律风险不可逆，强制驳回至少1次 |
| 系统挂了 | `/591 诊断` | 医部优先，6步诊断框架快速定位根因 |
| 季度复盘 | `/591 进化` | 手动触发进院分析，优化系统 |

### 2.3 理解输出结构

591 的最终交付包含：

| 文档 | 谁写的 | 内容 |
|------|--------|------|
| 问题定义书 | 察省 | 你的需求被精确翻译成可验证的目标 |
| 方案终稿 | 策省（经谏省审查） | 推荐方案，含风险矩阵和假设清单 |
| 审议结论 | 谏省 | 方案通过了哪些审查，哪些风险已缓解 |
| 执行产出 | 九部 | 具体交付物（分析报告/代码/文案/财务模型） |
| 验证报告 | 验省 | 实际结果 vs 成功标准的对照表 |
| 下一步行动 | 协省 | 你可以立即执行的 ≤3 条动作 |

### 2.4 学会「驳回」

作为天子，你可以在任何阶段叫停或要求修改：

- **察省理解有偏差** → 「察省，你没有理解我的真实需求，重新来」
- **策省方案太保守** → 「策省，我要一个更激进的方案」
- **谏省太松** → 「谏省，你再仔细审查一遍，驳回率太低了」
- **执行结果不满意** → 「验省，这里没有达到标准C，退回重做」

你就是天子，你可以随时调参。

## 三、场景示例

### 场景 1：评估一个新市场机会

```
/591 严审 评估进入沙特市场销售石油装备的可行性。
  当前我们在中东零客户。团队3人。预算是$50K试水。
  主要产品是FRP管道和井口设备。
```

**591 会做什么：**
- 察省：定义成功标准（3个月内至少1个付费客户？盈亏平衡时间？）
- 策省：给出3套进入策略（代理商→低价竞标→本地设点）
- 谏省：强制驳回至少1次（「代理商模式无法建立品牌，长期依赖第三方」）
- 兵部：沙特竞品矩阵 + 市场调研
- 学部：沙特石油装备监管/认证要求
- 刑部：制裁合规审查、沙特本地化率（iktva）要求
- 户部：$50K 预算的 12 个月现金流分析
- 验省：逐条验证 → 交付

### 场景 2：紧急故障排查

```
/591 诊断 我们的网站 cnps.com 突然无法访问，SSL证书报错。
  用户集中在欧洲，中国正常。最近没有做任何配置修改。
```

**591 会做什么：**
- 察省：收集症状（什么时间开始？影响范围？）
- 协省：路由医部优先
- 医部：6步诊断——
  1. 症状：SSL证书报错，欧洲用户，中国正常
  2. 时间线：是什么时候开始的？
  3. 分层排除：CDN节点→源站→证书链→DNS
  4. 假设验证：Cloudflare欧洲节点证书过期？
  5. 修复：临时止血+根治方案
  6. 验证：修复后确认
- 验省：确认问题解决
- 交付：故障根因报告 + 预防措施

### 场景 3：技术选型

```
/591 为一人公司选一个网站建站方案。
  需求：产品展示+博客+简单的联系表单。
  我自己会一点前端，但不想花太多时间维护。
  预算：$50/月以内。
```

**591 会做什么：**
- 察省：定义成功标准（上线时间/维护成本/可扩展性）
- 策省：Webflow vs Framer vs Next.js+MDX vs WordPress
- 谏省：「Webflow每月页面浏览量限制在10K，博客流量增长后会超；WordPress有安全漏洞频率最高的CMS记录」
- 工部：技术可行性评估 + 3档工期估算
- 户部：3年总成本对比（建站费+月费+维护时间折现）
- 礼部：品牌调性建议
- 验省：验证 → 交付推荐方案

### 场景 4：品牌定位

```
/591 帮我做一个完整的品牌定位方案。
  公司是做 AI Agent 外贸自动化的，叫 CNPS。
  目前没有品牌，客户不知道我们是干什么的。
  需要：品牌名建议 + Slogan + 一句话定位 + 品牌人格。
```

**591 会做什么：**
- 察省：定义成功标准（品牌名可注册+一句话让客户记住+与AI外贸差异化）
- 策省：3套品牌定位方向（专业/亲切/极客）
- 谏省：「方向B的『亲切』在B2B外贸市场容易被理解为『不专业』」
- 礼部：命名候选+Slogan+品牌人格+内容策略
- 刑部：商标可用性预检
- 兵部：竞品品牌定位矩阵
- 验省：验证 → 交付

## 四、每个 Agent 的专长

### 五省——你的智囊团

| 角色 | 什么时候找它 | 它会给你什么 |
|------|-------------|-------------|
| **察省** | 你不确定自己要解决什么问题 | 把模糊想法翻译成精确目标 |
| **策省** | 你需要多种选择来比较 | ≥3 套方案 + 假设 + 风险 |
| **谏省** | 你想知道方案会不会失败 | 挑出每个方案的致命缺陷 |
| **协省** | 你需要有人整体负责交付 | 拆任务、分派、追踪、汇总 |
| **验省** | 你想确认真的做到了 | 结果 vs 目标的逐条对照 |

### 九部——你的执行团队

| 部门 | 专长 | 一句话调它 |
|------|------|-----------|
| 兵部 | 市场、竞品、获客 | 「帮我分析竞争对手」 |
| 吏部 | 招聘、团队、管理 | 「帮我写一个 JD」 |
| 礼部 | 品牌、文案、内容 | 「帮我写品牌故事」 |
| 户部 | 财务、数据、ROI | 「帮我算这笔投资值不值」 |
| 刑部 | 法律、合规 | 「帮我审查这份合同的风险」 |
| 工部 | 代码、架构、技术 | 「帮我做技术选型」 |
| 学部 | 研究、学习 | 「帮我研究这个新领域」 |
| 医部 | 诊断、修 Bug | 「帮我查这个故障的根因」 |
| 创部 | 创新、头脑风暴 | 「帮我想10个完全不同的方向」 |

### 进院——你的进化引擎

| 功能 | 作用 |
|------|------|
| 绩效追踪 | 每次任务完成后的质量评分 |
| 知识结晶 | 从成功/失败中提取可复用的模式 |
| 能力发现 | 找出团队还缺什么能力 |
| 提示词进化 | 让 Agent 越用越好 |

## 五、常见问题

### Q：591 和直接问 Claude 有什么区别？

直接问 Claude = 请一个聪明人给建议。
用 591 = 请 15 个专业人士组成团队，按流程运作。

关键差异：
- 直接问：你可能得到一个漂亮的答案，但没人审查过它。
- 591：一个 Agent 提案，另一个 Agent 拆台，第三个 Agent 执行，第四个 Agent 验证。

### Q：什么任务不适合 591？

- 「今天天气怎么样？」——单轮问题，直接用 Claude。
- 「帮我打开这个文件」——纯操作，不需要分析。
- 「我情绪不好，想聊聊天」——关系型对话，不是解题型对话。

### Q：591 会犯错吗？

会。但它有三层纠错机制：
1. 谏省在方案阶段驳回错误方案
2. 验省在执行后验证结果
3. 进院在事后复盘提炼教训

它比「一个聪明人给建议」的错误率低，但不是零。

### Q：我需要记住所有 Agent 吗？

不需要。你只需要记住四个命令：`/591` `/591 极速` `/591 严审` `/591 诊断`。589 会自动判断调用哪些 Agent。

### Q：怎么让 591 越用越好？

1. 每次任务完成后，说 `「进院，记录这次任务」`
2. 定期说 `/591 进化`
3. 发现某个 Agent 的产出质量下降时，告诉操作者检查该 Agent 的 System Prompt

## 六、天子决策清单

作为天子，你需要在以下节点做决策：

| 节点 | 决策 | 不决策的后果 |
|------|------|-------------|
| 察省完成后 | 确认问题定义正确？成功标准合理？ | 整个团队在错的方向上努力 |
| 策省+谏省完成后 | 从 ≥3 套方案中选 1 套 | 没有选择权 = 被剥夺决策权 |
| 协省分派后 | 确认参与部门合理？ | 不该参与的人参与了，该参与的人没参与 |
| 验省不通过时 | 接受偏差 or 退回重做？ | 带着已知缺陷交付 |
| 交付后 | 批准进院记录 | 错误可能在下一次重复 |

## 七、从第一性原理的一句话总结

```
你面对的任何问题 = 从「现在在哪」到「要去哪里」的未知路径。

591 做七件事：
  察（看现在在哪、要去哪里）
  策（想怎么去——至少三条路）
  谏（驳——每条路哪会死人）
  协（分——谁走哪段）
  做（九部执行）
  验（查——真的到了吗）
  进（学——下次怎么更快）

这七件事是解决任何问题的充分必要条件。
```
